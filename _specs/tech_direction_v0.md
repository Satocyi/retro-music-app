# 技術方向ドキュメント v0

この文書は、**Phase 2（技術スタック決定）に入る前に**、プロジェクトの技術方向とガードレールを固定するための資料である。

- 入力イベントの処理手順や具体的な API シグネチャ、プロジェクト構成の細部など **実装詳細は書かない**。
- 本アプリの価値の中心は一般的な音楽管理ではなく、`_specs/wheel_spec_v0.md` で固定した **触感品質**（ホイール触感、haptic、tick 音とタッチの同期、微妙な慣性・微遅延、工業製品感、「機械を触っている」感覚）である。技術選定はこの優先順位に従う。
- `_docs/SYSTEM_CORE.md`、`_specs/wheel_spec_v0.md`、`_specs/mvp_flow_v0.md` と矛盾させない。

**バージョン**: v0（初版）  
**適用**: MVP フェーズおよび Phase 2 以降の技術議論のたたき台

---

## 1. このドキュメントの目的

1. **技術スタックの「詳細決定」を先送りせずとも**、実装側・複数 AI がぶれないよう、**採用する方向／採用しない方向** を文章で固定する。
2. 触感実装が **クロスプラットフォーム枠組みや重いバックエンド依存** に飲まれないよう、早期に **許容する妥協と、やってはいけない選択** を明示する。
3. 将来的に MusicKit 等へ拡張しうるにもかかわらず、**MVP では音楽再生をモックに留める** という経済性（開発コスト対体感検証の ROI）を、技術判断として記録しておく。
4. App Store での公開を将来視野に入れたときに効く **法務・審査・プライバシー上のリマインダー** を、実装とは切り離したチェックリスト水準で置く。
5. 正本ファイル（`_docs/SYSTEM_CORE.md` 等）ではなく **_specs 配下の作業資料**として、必要ならバージョンアップや差し替えができるようにする。

---

## 2. 採用する技術方向（MVP 前提）

| 領域 | 方向 |
|------|------|
| **対象プラットフォーム** | **iOS 専用**で開始する。Android 同時開発は行わない。 |
| **UI フレームワーク** | **SwiftUI を中心**に画面・状態遷移を組み立てる。 |
| **SwiftUI と UIKit の関係** | 原則 SwiftUI。**必要時のみ** UIKit（または UIViewRepresentable / UIHostingController 等の橋渡し）で補助する。「UIKit に逃げ続ける」を避ける。 |
| **触覚** | **CoreHaptics は MVP ですぐ必須としない**。ただし `wheel_spec_v0` に沿った haptic は体験の核のため、**将来の第一候補として位置づけ**、Early Phase での実機検証の対象としてよい（導入タイミングは別判断可）。|
| **音楽再生** | MVP は **モック再生**（ダミータイムライン／プレースホルダー音声・操作と同期した「再生状態」だけ等）で足りる。**Apple Music / Spotify 連携は MVP 対象外**。 |
| **将来のネイティブ音楽 API** | 設計概念として **MusicKit を差し込める受け口**（抽象レイヤ・アダプタ方針）だけ残す。**MVP で MusicKit に依存しない**。 |
| **状態・データ** | **ローカル状態中心**。オフラインで完結する範囲から始める。 |
| **外部バックエンド** | MVP では **Firebase や同等の総合 BaaS** のような「アカウント・分析・ログ・Crash・Remote Config を束ねた重依存」を **前提にしない**。 |

---

## 3. 採用しない技術方向（MVP における「やらない」）

- **クロスプラットフォーム・アプリフレームワークでの MVP**：**React Native / Flutter は MVP では採用しない**。（触感・低遅延同期・プラットフォーム固有の細部への素直さ／保守コストの観点で、現時点の優先順位と合わない。）
- **Android 同時立ち上げ**：リソース分散と体感チューニングの二分を避ける。
- **MVP でのストリーミングサービス統合**：Apple Music / Spotify 等のログイン・カタログ・再生権。
- **MVP でのクラウド主導のアーキテクチャ**：ユーザーデータをクラウドに集約することが前提の設計（必要になったフェーズでのみ再検討）。
- **体験核を損なう「とりあえず便利」の技術選択**：一覧最適化のためだけのスクロール基盤差し替え、触感と無関係な巨大ライブラリ導入など。

---

## 4. SwiftUI 中心にする理由

1. **MVP は iOS のみ**。単一プラットフォームであれば、抽象化コストより **プラットフォーム標準への寄与** が触感チューニングのサイクルを短くしやすい。
2. 宣言的 UI と状態バインディングは、**「操作 → 状態 → 視覚・聴覚・触覚」**という同期を追うのに適しており、`wheel_spec_v0` の **tick とタッチの同期**・**止まりの体感** と相性がよい。
3. MVP の画面数・機能範囲は限定的であり、SwiftUI で足りない部分は **補助的に UIKit** で埋める方が、クロスプラット全体を引きずるより **変更コスト（学習・デバッグ・レビューの往復）** が低い見込み。
4. 将来 MusicKit と統合するときも、**Swift / Apple スタックと同一経路**であり、プラットフォーム境界上のギャップが少ない。

---

## 5. UIKit 補助を許容する理由

1. SwiftUI のみでは **細かなジェスチャー競合・描画サイクル・既存/UIKit ウィジェットの再利用** が難しい箇所が出うる。そのとき **体感を落とさない** ことを優先し、UIKit を **局所適用** する。
2. **「UIKit への全面退行」ではない**。採用判断の基準は「SwiftUI で同等の質感・遅延が取れず、体感の核に影響するか」であり、許容するのは **補助線** とする。
3. ドキュメント上の約束として **「SwiftUI が主、UIKit はピンポイント」** と書いておけば、AI による無秩序な構成分裂を抑制できる。

---

## 6. CoreHaptics・Audio・gesture の考え方

**共通の原則**：いずれも **主役は「コンテンツ消費の快適さ」ではなく、状態が一段進んだことを身体で読む読み取り線**（`wheel_spec_v0` の haptic／tick と整合）。

| 論点 | 考え方 |
|------|--------|
| **CoreHaptics** | **将来利用候補**。MVP で未採用でもよいが、体感検証フェーズでは **機械的反発・短く線形な感触** が取れるかを見る価値は高い。**レトロ玩具化・振幅競争は禁止**のまま適用する。デバイス非搭載時はフェイルソフト前提。|
| **Audio（tick・UI 音）** | 同期の正確さが体験価値に直結する。**音楽本編と競合しない**範囲で、レイテンシを意識した設計（詳細アルゴリズムは実装フェーズ）。 |
| **Gesture** | スクロール最適解のジェスチャーに全面寄せしない。**質量・遅れ・止まり** を優先したカスタム挙動を許す。プラットフォーム標準のみで「無限リスト感」になる場合は、仕様側で再検証する。 |

---

## 7. モック再生から始める理由

1. **ROI**：MVP で検証すべき第一価値は **触感・選択前の時間・世界観** である。著作権・契約・カタログ品質という不確実性が高い「本再生」に先に拘束されると、**触っていない週間** が増え、体感改善のサイクルが細る。
2. **リスク回避**：ユーザー提供音源ポリシー・外部 API の条件は **SYSTEM_CORE が禁じる AI 単独決定域** と重なる。**モックにより法務論点と切り離し**、開発を進められる。
3. **MusicKit は後でも差し替えやすくする**：別節「受け口」で、**再生ソースを差し替え可能**と定義することで、「モック → ネイティブ」の移行経路のみを約束する。

---

## 8. 将来 MusicKit を差し込むための受け口（設計レベル）

実装コードではなく、**守るべき境界の言葉**として以下を約束する。

1. **`Playback*` / `MusicLibrary*` などの具象 API を UI とホイール直結にしない**。「何を選べるか」「何が再生状態か」を表す **薄いモデルとプロトコル（または準同等の抽象）** を一度挟む。
2. **キュー・現在トラック・再生位置・許可状態** は、将来的に SDK が供する情報へマッピングできる形でモデル化する。**UI は「ソースがモックか MusicKit か」を知らない**のが理想。
3. **認証・アカウント依存の画面フロー**は MVP で作り込まないが、将来的に **`Authorization` に相当する状態** が差し込まれることを阻害しない分割を心がける。
4. 「受け口を残す」は **機能フラグだけを散らすこと** と等価ではない。**データの流れが一本化されていること** が目的。

詳細クラス名・モジュール分割は実装フェーズで決める。本書は **存在すべき論理境界**のみ固定する。

---

## 9. MVP で避ける外部依存

- **総合クラウド基盤**（典型例：Firebase が提供する広範囲プロダクト群への一体的依存）
- **体験核と無関係な広範ユーティリティの山**（単一機能のために大きなフレームワークを引くこと）
- **ストリーミングサービスの公式／非公式 SDK**：MVP スコープ外
- **分析・広告ネットワークの常時オン**：課金・プライバシーは別途サトシ判断（`SYSTEM_CORE`）
- 「とりあえず入れられる」だけの **クラッシュレポートのみに見えても運用ポリシーが決まっていない**外部サービス

## 10. App Store 商用化を見据えた注意（チェックリスト水準）

**法的・運用上の決定はサトシおよび専門家の領域**である。以下は開発が迷子にならないためのリマインダー。

| 項目 | 注意 |
|------|------|
| **権利・音源** | 実音源やカタログ利用を広げる段階で、**ライセンス・利用規約のレビュー**が必須。MVP のモックはその境界をまたがないことを前提とする。 |
| **Privacy Nutrition / 収集データ** | 将来ネットワーク・ログ・アナリティクスを入れるとき、**収集項目と目的の明示**が必要になりうる。いまはローカル中心で複雑化を避ける。 |
| **アプリ機能と審査** | MusicKit／外部サービスを入れた時点で、**機能説明と実態の一致**、必要な Disclosure の有無が論点になりうる。 |
| **感覚的機能（haptics・音声）** | ユーザー環境により挙動差がある。**「ない端末での劣化」を許容**し、体感の約束文言を売り文句にしない（誤解による減点を防ぐ）。 |
| **サブスク・課金** | 課金は `SYSTEM_CORE` で AI が決めない。導入時は税務・表示・リストア要件が別問題として発生する。 |

---

## 11. AI が勝手に追加しない技術

以下は、`AGENTS.md` / `SYSTEM_CORE.md` と整合させた **追加禁止・要確認リスト**として扱う（重複は意図的）。

1. MVP 対象外の **Android ネイティブ／Flutter／React Native** 前提の構成や依存
2. **Apple Music / Spotify** を前提にしたログイン・API・OAuth フロー
3. **Firebase 等クラウド基盤**を「なんでも入れ」の形で標準インフラ化すること
4. **体験コンセプト・UX の核・課金・広告モデル・配布方針** に触れる技術（例：特定の広告 SDK）
5. **音楽ファイルの取得・ホスティング・著作権処理** に踏み込む実装詳細や外部サービス
6. 「便利だから」という理由だけの **大規模な状態管理ライブラリ・DI コンテナ・アーキテクチャお祭り**

## 12. 【判断が必要】技術変更としてサトシに確認すべきもの

実装フェーズでの具体的選定でも、論点になったら必ず **`【判断が必要】`** に上げて止めること。

| 論点 | 触れる理由 |
|------|-------------|
| **対象 OS の追加**（Android など）または **クロスプラットフォームへの移行** | UX・コスト構造・工期の全面的な変更 |
| **この文書との矛盾する技術スタック変更**（例：MVP で Flutter に移る） | ドキュメント正と実装の乖離 |
| **MusicKit／ストリーミング／ユーザー音源／クラウド同期** の採否と範囲 | 権利・プライバシー・審査 |
| **CoreHaptics の MVP 組み込み時期・必須化** | スコープ・スケジュール・実機要件 |
| **外部ライブラリの導入**（用途・サイズ問わず） | `SYSTEM_CORE` と `AGENTS.md` のルール |
| **匿名化されていないユーザー行動の収集・アナリティクス永続送信** | プライバシー方針 |
| **公開ストアでの課金・サブスク・アドネットワーク** | ビジネスモデル |
