## 0. この指示書の位置づけ

本書は「**SwiftUI で筐体〜ボタンの見た目を再描画しない**」ことを前提とする。

## 5. UIレイヤ構造（推奨の組み立て）

以下のような **論理スタックのみ**規定する。ファイル名／型名は実装側で調整してよい。

```
ZStack(alignment: .center)
 ├─ 背景（単色または既存のアプリBG方針に合わせる）
 ├─ deviceImage        // UIImage/SwiftUI Image — レイアウトの基準サイズとなる
 └─ InteractionLayer // Color.clear を重ねた領域 + gesture
```

**原則**：

1. **`deviceImage` がレイアウトのソースオブトラス**  
   「画像の論理サイズ・アスペクト比」を親が維持し、その **同一サイズ・同一原点** に `InteractionLayer` を重ねる。

2. **`InteractionLayer` は `allowsHitTesting(true)`**。  
   画像自体はヒットしないか、レイヤのみがヒットでもよい。**二重入力**を避ける。

3. **Safe Area**  
   セーフエリア外まで画面をどう張るかは `tech_direction_v0` の体感方針に従う。本書では **論理サイズとの一致検証優先**。余白調整係数 `-- safeInsets` は定数として切り離す。

## 6. ヒット領域の指定方法（調整ROIを最大化）

### 6-1. 正規化座標での定義（必須とする）

各ヒット矩形は、`deviceImage の表示サイズ を (1.0 × 1.0)` としたときの **相対矩形** で定義すること：

```
struct NormalizedHitRect {
    var origin: CGPoint   // x,y は 0...1 の範囲（左上原点）
    var size: CGSize      // width,height は比率
}
```

**理由**: 異なる機種サイズ・将来のレイアウト変更に対して、**係数のみの変更**で追従でき、レビューの差分が説明しやすい。

### 6-2. v0 で必須のキー領域名

次の名前を **名前付きで** holding すること（係数のみ更新すれば並べられる）：

| 名前 | 推定ターゲット | 許容入力 |
|------|----------------|-----------|
| `menuHit` | 左下MENU物理キーのベタ塗領域 | `TapGesture` のみでも可 |
| `selectHit` | 中央「選択」 | 同上 |
| `generateHit` | 右「生成→」 | 同上 |
| `wheelDiskHit` | 外周〜ウェル〜中央ボタンの含む近似円 | `DragGesture` 優先。**タップのみ**許容でも可 |

**初期係数**：HTMLプレビューで使った `./_design/previews/` の比率を **ひな型** にしてよいが、SwiftUI側で **確定値として再確認**すること（HTML と Swift はレンダパイプラインが異なるため完全一致は保証されない）。

### 6-3. 入力の競合順位（最小規定）

ホイール上で **ドラッグとタップが競合**する場合：

- **ドラッグ優先**。`minimumDistance > おおよそ8pt` のようにし、意図しないタップを抑止するのが望ましい。

理由: **誤タップ率**は操作感の主観評価に直結し、しきい値1本で **有意に改善しうる**。

## 8. 既存コードとの接続ポリシー（衝突回避）

`_specs/cursor_impl_visual_v1.md` は **SwiftUIでの筐体べクター再現**を前提としており **本フェーズとは矛盾する**。

実装フェーズ開始時には次のどちらかで **競合が顕在化しない**状態にすること：

**方針A（推奨）**: **`Memory Tuner Shell` を表す `View`** を **新規**に切り、`ContentView` からのみ差し込む。

**方針B**: 既存の `WheelView.swift` の **視覚子ツリーを一時封印**する（ロジック層だけ再利用）。

いずれの場合も、**モデル変更は不要**であり、モデル側に **UI状態を足さない**（判断が必要な設計論点はサトシ承認）。
