# SYSTEM_CORE

この文書は、このアプリ開発プロジェクトの「不変の原則」です。  
技術・UI・機能が変わっても、ここは原則として変更しません。

---

## 0. ミッション

このアプリは、単なる音楽再生アプリではなく、  
昔のデジタル音楽体験を現代のスマートフォン上で再構成するアプリである。

目的は、懐かしさ・手触り・偶然性・小さな発見を通じて、  
ユーザーが音楽を「便利に消費する」のではなく、「少し特別な体験として楽しむ」こと。

---

## 1. 体験の核

このアプリの核は以下とする。

- レトロな操作感
- 触って楽しいUI
- 過剰に便利すぎない体験
- 偶然出会う音楽
- 小さな演出による没入感
- 現代アプリの速さより、記憶に残る体験を優先する

---

## 2. やらないこと

このアプリは以下を目指さない。

- Spotify / Apple Music の代替
- 高機能な音楽管理アプリ
- 完全なレトロ機器の再現
- 複雑なSNS機能
- 最初から大規模な課金設計
- 最初から多数機能を詰め込むこと

---

## 3. MVPの考え方

最初のMVPでは、完成度よりも「触った瞬間に面白い」と感じる体験を優先する。

MVPで重視するもの：

- 起動した瞬間の世界観
- 音楽を選ぶ操作の楽しさ
- レトロUIの手触り
- 最低限の再生体験
- 1つか2つの象徴的な演出

MVPで後回しにしてよいもの：

- 完璧なライブラリ管理
- 高度な検索
- 複雑な設定画面
- 大量のテーマ
- 完璧なプレイリスト機能

---

## 4. AIが勝手に変えてはいけないもの

以下はAIが独断で変更してはいけない。

- アプリの体験コンセプト
- UXの核となる操作方法
- 課金・広告・ビジネスモデル
- 外部APIや外部サービスの追加
- ユーザーデータの保存方針
- 音楽ファイル・著作権に関わる扱い
- アプリ名・ブランド名
- 配布方針

該当する場合は、必ず「判断が必要」としてサトシに確認する。

---

## 5. 複数AIの役割

基本的な役割分担は以下とする。

| 役割 | 担当 |
|---|---|
| 最終判断 | サトシ |
| 設計・方針整理 | Claude |
| アイデア出し・構造化・文書化 | ChatGPT |
| 実装 | Cursor |
| 環境・技術検証 | 必要に応じてGemini等 |

AIは自分の役割を越えて勝手に決定しない。

---

## 6. 開発の進め方

開発は以下の原則で進める。

- 1ステップずつ進める
- 大きな変更を一度に入れない
- 実装前に目的を確認する
- 変更後は必ず動作確認する
- 迷ったら機能追加ではなく、体験の核に戻る
- 便利さよりも、このアプリらしさを優先する

---

## 7. ドキュメント運用

このプロジェクトでは以下を使う。

| ファイル | 役割 |
|---|---|
| SYSTEM_CORE.md | 不変の原則 |
| SYSTEM_CURRENT.md | 現在地・実装状態 |
| COLLABORATION_PROTOCOL.md | AI作業ルール |
| roadmapforAI.md | 今後の課題・優先順位 |
| Handover/ai_handover/ | セッション引き継ぎ |

作業開始時は、Core → Current → Protocol → roadmap → 最新handover の順に読む。

---

## 8. Handoverルール

Handoverには以下を書く。

- 何をやったか
- なぜそうしたか
- 何をやらなかったか
- 次にやること
- 触ってはいけないもの
- 未確認事項

推測や願望は書かない。  
不明なことは「不明」と書く。

---

## 9. 判断原則

迷った場合は、以下の順で判断する。

1. このアプリの体験の核に合っているか
2. MVPに必要か
3. 今やるべきか
4. 実装コストに対して体験価値があるか
5. 将来の変更を妨げないか

判断できない場合は、実装せずサトシに確認する。
