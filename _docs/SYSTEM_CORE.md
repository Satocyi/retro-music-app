# SYSTEM_CORE

この文書は、このアプリ開発プロジェクトの「不変の原則」です。  
技術・UI・機能が変わっても、ここは原則として変更しません。

---

## 0. ミッション

このアプリは、単なる写真フィルターアプリではない。
写真を速く加工するアプリでもない。

昔の携帯デジタル機器の手触りを借りて、iPhone写真を「記憶の空気」に変換する体験を作る。

ホイールで年代・季節・時間帯・天気・質感を選ぶ。
その選択過程そのものを、記憶をチューニングする体験にする。

目的は、懐かしさ・手触り・偶然性・小さな発見を通じて、  
ユーザーが写真を「便利に加工する」のではなく、「記憶の空気に変換する体験として楽しむ」こと。

---

## 1. 体験の核

このアプリの核は以下とする。

- レトロな操作感
- 写真の空気をチューニングする楽しさ
- 過剰に便利すぎない体験
- 偶然出会う質感・色味・記憶感
- 小さな演出による没入感
- 現代アプリの速さより、記憶に残る変換体験を優先する

---

## 2. やらないこと

このアプリは以下を目指さない。

- 高機能な写真編集アプリ
- Photoshop / Lightroom の代替
- SNS投稿専用の量産フィルターアプリ
- AI画像生成を最初から全面採用すること
- 完全なレトロ機器の再現
- 複雑なSNS機能
- 最初から大規模な課金設計
- 最初から多数機能を詰め込むこと

---

## 3. MVPの考え方

最初のMVPでは、完成度よりも「触った瞬間に面白い」と感じる体験を優先する。

MVPで重視するもの：

- 起動した瞬間の世界観
- 写真を選ぶ / 撮る体験
- ホイールで空気を調整する操作感
- レトロUIの手触り
- 低コストな写真フィルター変換
- 1つか2つの象徴的な演出

MVPで後回しにしてよいもの：

- AI画像生成API
- 高度な写真編集
- レイヤー編集
- 大量のフィルター
- SNS機能
- 課金実装
- サーバー保存

---

## 4. AIが勝手に変えてはいけないもの

以下はAIが独断で変更してはいけない。

- アプリの体験コンセプト
- UXの核となる操作方法
- 課金・広告・ビジネスモデル
- 外部APIや外部サービスの追加
- ユーザー写真データの保存方針
- 写真のサーバー送信有無
- AI画像生成APIの採用
- アプリ名・ブランド名
- 配布方針

該当する場合は、必ず「判断が必要」としてサトシに確認する。

---

## 5. 複数AIの役割

基本的な役割分担は以下とする。

| 役割 | 担当 |
|---|---|
| 最終判断 | サトシ |
| 設計・方針整理 | Claude |
| アイデア出し・構造化・文書化 | ChatGPT |
| 実装 | Cursor |
| 環境・技術検証 | 必要に応じてGemini等 |

AIは自分の役割を越えて勝手に決定しない。

---

## 6. 開発の進め方

開発は以下の原則で進める。

- 1ステップずつ進める
- 大きな変更を一度に入れない
- 実装前に目的を確認する
- 変更後は必ず動作確認する
- 迷ったら機能追加ではなく、体験の核に戻る
- 便利さよりも、このアプリらしさを優先する

---

## 7. ドキュメント運用

このプロジェクトでは以下を使う。

| ファイル | 役割 |
|---|---|
| SYSTEM_CORE.md | 不変の原則 |
| SYSTEM_CURRENT.md | 現在地・実装状態 |
| COLLABORATION_PROTOCOL.md | AI作業ルール |
| roadmapforAI.md | 今後の課題・優先順位 |
| Handover/ai_handover/ | セッション引き継ぎ |

作業開始時は、Core → Current → Protocol → roadmap → 最新handover の順に読む。

---

## 8. Handoverルール

Handoverには以下を書く。

- 何をやったか
- なぜそうしたか
- 何をやらなかったか
- 次にやること
- 触ってはいけないもの
- 未確認事項

推測や願望は書かない。  
不明なことは「不明」と書く。

---

## 9. 判断原則

迷った場合は、以下の順で判断する。

1. このアプリの体験の核に合っているか
2. MVPに必要か
3. 今やるべきか
4. 実装コストに対して体験価値があるか
5. 将来の変更を妨げないか

判断できない場合は、実装せずサトシに確認する。
