実装アイデアを何でも詰め込む場所ではなく、AIが次に何をすべきか迷わないための作業地図です。

## Phase 1: MVP体験設計

目的：実装前に「何を作るアプリか」を絞る。

- [ ] MVPで体験させたいことを1文で定義する
- [ ] レトロUIの中心モチーフを決める
- [ ] 最初の画面構成を決める
- [ ] 音楽再生を本物の音源で行うか、モックで始めるか決める
- [ ] MVPに入れる機能 / 入れない機能を分ける

---

## Phase 2: 技術スタック決定

目的：MVPに必要な技術だけを選ぶ。

- [ ] 対象OSを決める
- [ ] 開発技術を決める
- [ ] 音楽再生方式を決める
- [ ] ローカル音源 / サンプル音源 / 外部API の扱いを決める
- [ ] 外部ライブラリ利用方針を決める

---

## Phase 3: プロトタイプ実装

目的：最小限の「触って面白い」体験を作る。

- [ ] 起動画面を作る
- [ ] レトロUIの中心操作を作る
- [ ] 曲選択のモックを作る
- [ ] 再生 / 停止の基本操作を作る
- [ ] 1つ目の象徴的な演出を入れる
- [ ] 実機またはローカルで動作確認する

---

## Phase 4: 実機テスト・改善

目的：触った感覚を見て改善する。

- [ ] 操作感を確認する
- [ ] レトロ感が出ているか確認する
- [ ] 過剰に便利になりすぎていないか確認する
- [ ] MVPに不要な機能を削る
- [ ] 次バージョンで追加する機能を分ける

---

## 人間判断が必要な論点

以下はAIが勝手に決めない。

- アプリ名
- 対象OS
- 技術スタック
- UXの核
- 音楽ファイルの扱い
- 外部API利用
- 課金・広告
- 配布方針
- ブランド表現

---

## 今はやらないこと

- 本格的な音楽管理機能
- 複雑なSNS機能
- 多数のテーマ作成
- 課金実装
- 外部音楽サービス連携
- 大規模な設定画面
- 完璧なプレイリスト機能

## 更新ルール

- Phaseが進んだら更新する
- 完了した項目は `[x]` にする
- 未確定案は書かない
- 新しいアイデアは、必要なら `_design/` や `_specs/` に分けて記録する
- SYSTEM_CURRENT.md と矛盾しないようにする
